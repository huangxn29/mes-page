# Code Review 规范

## Review 流程

1. **开发者**创建 PR → 指派 Reviewers → 标记为 Ready for Review
2. **Reviewer** 在 24 小时内开始审查
3. **Reviewer** 提出意见 → 开发者回应并修改
4. **至少 1 人 Approve** 后方可合并

## 检查维度

| 维度 | 检查要点 |
|------|----------|
| **正确性** | 逻辑是否正确？边界条件是否处理？是否存在 NPE？ |
| **安全性** | 输入校验？XSS 风险？权限检查？ |
| **性能** | 不必要的重渲染？大列表优化？缓存合理？ |
| **可维护性** | 代码清晰？有适当注释？遵循现有风格？ |
| **测试覆盖** | 有充分的单元测试？覆盖边界情况和异常路径？ |
| **一致性** | 遵循项目架构模式？组件设计一致？ |

## 评论分级

| 级别 | 含义 |
|------|------|
| ❌ **REQUIRED** | Bug/安全/严重逻辑错误。未修复不得合并 |
| ⚠️ **SUGGESTION** | 可改进的地方，开发者判断是否采纳 |
| ❓ **QUESTION** | 对代码有疑问，需开发者解释 |
| 👍 **NICE** | 对好的实现给予肯定 |

## 最佳实践

- **Reviewer**: 关注逻辑和架构，而非代码风格（由 Linter 保证）
- **Reviewer**: 问问题而非下结论 — "这个方法可能返回 null 吗？"
- **开发者**: 对每条评论回复（"已修复"、"有道理"、"这里是因为..."）
- **速度优先**: 小 PR 优先审查，避免阻塞下游
